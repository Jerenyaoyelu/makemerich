# 多源自动采集实施计划（回家执行版）

## 1. 目标

把当前“单一东财接口 + 手动兜底”升级为“多源自动采集 + 标准化 + 失败可观测”的稳定数据层。

---

## 2. 核心方案

不是单纯做爬虫，而是做一套工程化链路：

1. 多数据源采集（主源 + 备源）
2. 统一字段标准化（策略层不感知数据源差异）
3. 自动重试与降级（失败可追踪）
4. 本地快照与日志（可追溯）
5. UI 展示运行状态（当前数据源、失败原因、连续失败次数）

---

## 3. 目标架构

## 3.1 数据源优先级

- S1：Eastmoney（快，字段较全）
- S2：Sina/Tencent（备用）
- S3：Tushare（规范，可能有额度限制）
- S4：本地快照（仅手动选择，不自动回退）

执行逻辑：

- 自动采集时依次尝试 S1 -> S2 -> S3
- 成功即返回，并记录使用源
- 全失败则报错（不自动回退本地）

## 3.2 标准字段

各源最终输出统一字段：

- `symbol`
- `name`
- `theme`
- `pct_chg`
- `turnover`
- `amount`
- `volume_ratio`
- `amplitude`
- `theme_strength`
- `sector_linkage`
- `stock_strength`
- `capital_support`
- `risk_tag`
- `snapshot_time`
- `source`

---

## 4. 技术实现清单

## 4.1 `core/data_provider.py`

新增/改造：

1. `fetch_from_eastmoney()`
2. `fetch_from_sina_or_tencent()`
3. `fetch_from_tushare()`
4. `fetch_live_signals_multi_source()`（统一调度器）
5. `normalize_schema()`（字段映射与补齐）
6. `compute_features()`（计算四大评分因子）

要求：

- 每个 source 函数只负责“抓取+初步映射”
- 所有特征计算在同一个通用函数中完成

## 4.2 `app/main.py`

新增 UI 状态区：

1. 当前使用数据源
2. 最近成功时间
3. 最近失败原因
4. 连续失败次数
5. 手动加载本地缓存按钮（保留）

## 4.3 日志与快照

目录约定：

- `logs/fetch.log`：采集过程日志
- `data/raw/YYYYMMDD_HHMM_source.csv`：原始快照
- `data/latest_signals.csv`：最新标准化数据

记录内容：

- source
- 耗时
- 记录数
- 失败堆栈摘要

---

## 5. 网络与稳定性策略

1. 请求超时（例如 8-10 秒）
2. 重试次数（2-3 次，指数退避）
3. 随机抖动（避免固定节奏）
4. 代理与直连双模式尝试
5. 错误分类：`ProxyError` / `Timeout` / `SchemaError` / `HTTPError`

---

## 6. 两周落地排期

## 第 1 周（打通链路）

1. 抽象多源接口
2. 接入 S1 + S2
3. 完成统一字段映射与特征计算
4. 写入 `latest_signals.csv`

验收：

- 任一源可成功时，UI 能正常出候选
- 输出字段完整且无空关键列

## 第 2 周（稳定与可观测）

1. 加日志系统和 raw 快照
2. 加 UI 状态面板
3. 接入 S3（可选）
4. 做失败演练（断网、代理错误、字段变更）

验收：

- 失败时报错清晰，不静默
- 能从日志定位具体失败源与原因

---

## 7. 回家后开工步骤（按顺序）

1. 拉最新代码并运行当前项目
2. 新建 `logs/` 目录
3. 在 `data_provider.py` 先拆出 source 函数骨架
4. 先接 S2（作为东财失败备源）
5. 写统一 `normalize_schema()`
6. 在 UI 加“当前数据源/失败原因”显示
7. 跑一次全流程并留存首份日志

---

## 8. 成功标准（Done Definition）

满足以下全部才算完成：

1. 自动采集不再单点依赖东财
2. 失败时明确报错且可定位
3. UI 可看到数据源与健康状态
4. 策略层只依赖统一字段，不依赖具体来源
5. 你可在 5 分钟内判断“今天能不能用这套数据交易辅助”
